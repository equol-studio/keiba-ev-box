# keiba-app 方向性設計書（CEO策定）

- 作成: 2026-07-05 / 起案: CEO(advisor)
- 対象: 既存PWA「競馬 期待値ボックス」(GitHub: Terea-equol/keiba-ev-box / 公開: equol-studio.github.io/keiba-ev-box)
- 前提: 司令官=現役PT・馬券戦略オーナー。JRAのEVレースだけ買う。基本は3連複5頭BOX＋妙味券種。
- 本書はあくまで方向性の合意用。司令官の「作成して」まで実装はしない。

---

## 0. 結論（最初に読む3行）
- **推奨: 「AIによるEV選出 × 人間の最終判断」のハイブリッド支援ツールに進化させる。**
- **オッズは"使う"。ただし基準は「事前予想オッズ vs 自前の勝率見立て」の乖離（＝妙味）検出。確定オッズは枠順後の最終チェックに限定。**
- **claude-in-chrome は"分析役（頭脳）"、GitHub Actions は"下ごしらえ役（配膳）"に役割分担。全自動化は狙わない。**

---

## 1. コンセプト定義 — このアプリは何者か

**一言: 「買うレースと買う馬を、EV(期待値)の観点で絞り込む"意思決定支援ツール"」。**

- 予想を"当てる"アプリではなく、**-EVのレースを捨て、+EVの妙味だけ拾う判断を高速化する**ツール。
- 司令官の既存ワークフロー（EVレースだけ／3連複5頭BOX／netkeiba出典／枠順確定後に最終化）を**置き換えず、増幅する**。
- 立ち位置の明確化:
  - やること: レース取捨（参戦/見送り）、妙味馬の抽出、券種の当てはめ、買い目の提示。
  - やらないこと: 全レース予想、自動購入、的中保証。**「見送り」を堂々と出せることが価値**。
- 差別化の核: 一般の予想サイトは「本命を当てる」設計。本アプリは**「控除率に負けない妙味があるか」だけを見る逆張り設計**。ここは既存の強み（オッズ一辺倒を避ける思想）と一致。

## 2. EV(期待値)の定義と算出アプローチ

### 2-1. EVの定義（本アプリの共通言語）
```
EV(期待値) = (自分が見積もる的中確率 p) × (配当オッズ o) − 1
EV > 0 なら「妙味あり=買う候補」、EV ≤ 0 なら「見送り」
```
- 難所は **p（勝率/連対率/複勝率の見立て）**。o は市場（オッズ）から取れるが、p は作らないといけない。
- JRA控除率（払戻率の裏返し）を券種選択の前提に置く（出典: JRA）:
  - 単勝・複勝 20%（払戻80%）／ワイド 22.5%（77.5%）／馬連 22.5%／3連複 25%（75%）／3連単 27.5%。
  - **含意: 控除の軽い複勝・ワイドはEVを出しやすく、3連複は妙味(乖離)が大きい時だけ。** 現行data.jsの「割安1頭はワイド/複勝、絞れる時だけ3連複」という方針は控除率的に正しい。継続推奨。

### 2-2. オッズをどう扱うか（3つの選択肢）

| 案 | 内容 | 長所 | 短所 |
|---|---|---|---|
| A. オッズ非使用（現行） | 枠順・コース・構造だけで取捨 | 事故らない/実装済 | pもoも無いので"EV"を名乗れていない。妙味の定量化不可 |
| B. 事前予想オッズ基準（推奨） | netkeibaの想定/AI予想オッズを o とし、AIが見積もる p と比較して乖離＝妙味を検出 | 前日〜枠順段階で妙味を出せる／司令官の運用時間と合う | 予想オッズは変動する。あくまで一次スクリーニング |
| C. 確定オッズ基準 | 発走直前の確定オッズで最終EV判定 | 最も正確 | 直前しか使えず司令官の生活リズムに合わない。全レース張り付き不可 |

**推奨: B を主、C を従。**
- 前日〜枠順確定: **B（予想オッズ×AI見立て）で妙味レース・妙味馬を絞る**＝アプリの主戦場。
- 発走直前: **C（確定オッズ）は"最終確認の一手"**。司令官が現地/自宅で「本当に割安のまま残っているか」だけ照合。全レースやらない。

### 2-3. p（勝率見立て）の作り方 — 簡易ルール vs スコアリング

**推奨: 「スコアリング（0〜100の妙味スコア）」を段階導入。ただしMVPはルール継続。**

- **フェーズ0(既存)**: ルールベース取捨（新馬/未勝利・少頭数=見送り 等）。これは残す＝土台。
- **フェーズ1(MVP)**: claude-in-chrome が各妙味レースを読み、**馬ごとに"妙味スコア"を言語化＋数値化**（例: 枠/脚質適性/斤量/騎手/前走位置取り/馬場適性 の加点、市場人気に対する割安度）。
  - この段階では厳密な確率モデルは作らない。**「AIの構造分析 → 妙味スコア → 予想オッズとの乖離チェック」**で十分に現行を超える。
- **フェーズ2(将来)**: 過去データで p のキャリブレーション（当てた/外した記録＝2-4の検証ログ）を蓄積し、スコア→実勝率の対応表を作る。ここで初めて"本物のEV"に近づく。

**なぜスコアリング全面移行を今やらないか**: p の精度検証データが無い状態で数値EVを出すと、精緻に見えて根拠が空洞。まずAIの言語分析＋乖離検出で運用し、当たり外れログ(2-4)を貯めてから数値化する。

### 2-4. 検証ログ（EVの命綱）
- 毎開催、**「選出した妙味馬・券種・想定オッズ・結果・回収」を1行で記録**（別途 results.js 等）。
- これが無いと p は永遠に当てずっぽう。**フェーズ2の燃料**であり、司令官が「このアプリは勝てているか」を判断する唯一の根拠。MVPから軽量でも必ず入れる。

## 3. データ取得方式 — claude-in-chrome × GitHub Actions のすみ分け

**基本思想: 「機械的に取れるものは Actions、判断が要るものは claude-in-chrome」。**

| 工程 | 担当 | 理由 |
|---|---|---|
| 開催・出走一覧の取得、クラス/距離/頭数/枠順の抽出、ルール取捨(参戦/見送り下書き) | **GitHub Actions（既存 refresh.mjs 拡張）** | 定型・スクレイピングで足りる。既に動いている |
| 予想オッズ/想定オッズの取得 | **Actions（API `api_get_jra_odds.html` 等を検証のうえ）** | 定型取得。ただし取得可否・利用規約は要確認(→未決) |
| 妙味レースの深掘り分析（出馬表・過去走・馬場・展開を読み妙味馬とスコアを選出） | **claude-in-chrome（能動操作＝頭脳）** | ここが人/AIの判断領域。ルールでは出せない |
| 予想オッズと妙味スコアの乖離判定→買い目・券種の当てはめ | **claude-in-chrome（＋司令官レビュー）** | EVの意思決定そのもの |
| 枠順確定後の最終化、確定オッズでの直前チェック | **司令官＋claude-in-chrome** | 責任判断。自動化しない |
| PWAへの反映（data.js/auto.js 生成・push） | **Actions＋手動push併用** | 定型 |

- **すみ分けの要点**: 現行の「Actionsが全レース取捨の下書き(auto.js)」は**一次スクリーニングとして残す**。その上で claude-in-chrome が**"参戦"の出たレースだけ深掘り**し、data.js（本カード）を作る。二層構造は既に index.html が両対応（KEIBA_AUTO優先表示）しており、設計と実装が一致している＝流用可。

## 5. 画面/機能の方向性（残す/足す・MVP範囲）

### 現行から残すもの（資産として優秀）
- PWA構成・ダークな競馬場テーマUI・カード開閉・検索フィルタ（index.html は完成度高い、流用）。
- 「今週の方針／買い方の方針／全レース判定(scan)／複勝転がし」の情報設計。
- data.js を差し替えるだけの運用モデル（軽量で司令官が扱える）。
- KEIBA_AUTO(自動) を KEIBA_DATA(手動) より優先表示する二層構造。

### 足すもの（EV支援への進化）
1. **妙味バッジ**: 各妙味馬に「予想オッズ o / 見立て確率 p / EVの符号（+/−）」を表示。まずは +EV / 様子見 / -EV の3色バッジで十分。
2. **乖離の可視化**: 「市場人気○番人気なのにスコア上位＝割安」を一目で。現行の tag(軸/◎/○)にEV観点を接続。
3. **直前チェック用の最小オッズ枠**: 発走直前に確定オッズを手 or 半自動で1行だけ差し込める欄（現行の odds.rows を流用可）。
4. **検証ログ画面（results）**: 開催ごとの選出→結果→回収率。地味だが最重要（2-4）。

### MVPの範囲（最初のリリースで出すもの）
- **In**: 参戦レースの妙味馬＋妙味スコア(言語＋3色EVバッジ)、予想オッズ表示、買い目(現行の本線＋5頭BOX)、検証ログの器。
- **Out(後回し)**: 数値確率モデル、確定オッズ全自動取得、自動購入、複数場の全レース深掘り。
- 判断基準: **「司令官が土日にストレスなく回せて、勝ち負けを記録できる」ところまでがMVP**。

## 6. 部署編成の提案

CEOが全部抱えず、以下4部署に委任（各部署=必要時にサブエージェント起用）。

| 部署 | ミッション（1行） |
|---|---|
| **データ取得部** | netkeibaから開催/出馬表/枠順/予想オッズを安定取得し、機械可読データにする（Actions/refresh.mjs 拡張）。 |
| **EVロジック部** | 妙味スコアの定義とEV(乖離)判定ロジックを設計し、参戦レースの妙味馬・買い目を選出する（本アプリの頭脳）。 |
| **フロント部** | PWA(index.html)にEVバッジ・オッズ枠・検証ログを実装し、iPhoneで完結する体験を作る。 |
| **検証部** | 選出結果と回収率を記録・分析し、pの精度をキャリブレーションしてEVロジック部に改善を還元する。 |

- 当面は **データ取得部＋EVロジック部の2部が主役**。フロント部はMVP実装時、検証部はログ蓄積が貯まってから本格稼働。
- 引き継ぎ: 方向性確定後、要件定義は **director** へ。director が各部の作業チケットに分解。

## 7. 未決事項 → 司令官に確認すべき質問（厳選5）

1. **オッズ利用の踏み込み度**: 予想オッズ/確定オッズをアプリに取り込むのはOKか？（現行「オッズ非使用」思想からの転換。netkeiba/JRAの取得可否・規約リスクも要確認。ここが方向性の最大分岐）
2. **claude-in-chrome の運用者**: 深掘り分析のブラウザ操作は「司令官のPCで司令官が起動」か「秘書が代行」か？ 実行環境・ログインアカウントの前提を決めたい。
3. **対象範囲**: MVPは今まで通り「妙味レースだけ深掘り（メイン中心＋抽出平場）」でよいか？ それとも1開催全レースにEVスコアを付けたいか？（工数が大きく変わる）
4. **数値EVの見せ方**: 精度が未検証の段階で数値（例: EV+8%）まで出すか、当面は「+EV/様子見/-EV」の3段バッジに留めるか？（過信リスク vs 情報量）
5. **検証ログの粒度**: 回収率記録は「選出した妙味馬・券種・結果」の1行ログでよいか、賭け金・実払戻まで入れて実損益を追うか？（フェーズ2の精度を左右）
