## 0. 確定した方向性（最初に読む）
- **AIによるEV選出 × 人間の最終判断のハイブリッド支援ツールに進化させる。**
- **【決定】オッズは取り込む。予想オッズ＋確定オッズの両方を活用する。**（取得可否・規約リスクは要検討として §8 に明記）
- **【決定】claude-in-chrome は"秘書(Claude)が代行"して動かす。**実行環境・ログインアカウントは秘書運用前提で設計。
- **【決定】1開催"全レース"にEVスコアを付けたうえで絞り込む。**（妙味レースだけの深掘りではない）
- **【決定】数値EVまで出す。精度を重視し、スコアリング/検証にコストをかける。**
- **【決定】検証ログは実払戻・実損益まで追う。**（賭け金・実払戻を記録し回収率を可視化）

## 1. コンセプト定義 — このアプリは何者か

**一言: 「全レースをEVで採点し、+EVの妙味だけに賭ける判断を高速化する意思決定支援ツール」。**

- 予想を"当てる"アプリではなく、**-EVのレース/馬を捨て、+EVの妙味だけ拾う判断を高速化する**ツール。
- 司令官の既存ワークフロー（EVレースだけ／妙味券種／netkeiba出典／枠順確定後に最終化）を**置き換えず、全レース採点で網羅性を足して増幅する**。3連複5頭BOXは常設ではなく+EV時のみの任意提示（2026-07-05更新）。
- やること: 全レースのEV採点 → 参戦/見送りの取捨 → 妙味馬の抽出 → 券種の当てはめ → 買い目の提示 → 実損益の記録。
- やらないこと: 自動購入、的中保証。**「見送り」を堂々と出せることが価値**。
- 差別化の核: 一般の予想サイトは「本命を当てる」設計。本アプリは**「控除率に負ける-EVを機械的に排除し、市場との乖離(妙味)だけを拾う逆張り設計」**。

## 2. EV(期待値)の定義と算出【決定】

### 2-1. EVの定義（共通言語）
```
単勝EV     = p_win  × o_win  − 1
複勝/ワイド = p_hit  × o_pay  − 1
3連複      = p_combo × o_combo − 1
　p = 自前の見立て確率、o = 市場オッズ（予想 or 確定）
　EV > 0 = 妙味あり（買う候補） / EV ≤ 0 = 見送り
```
- JRA控除率を券種選択の前提に置く（出典: JRA）:
  - 単勝・複勝 20%（払戻80%）／ワイド・馬連 22.5%（77.5%）／3連複 25%（75%）／3連単 27.5%。
  - **含意: 控除の軽い複勝・ワイドはEVが出やすく、3連複は乖離が大きい時だけ。** 現行data.jsの券種方針（割安1頭はワイド/複勝、絞れる時だけ3連複）は控除率的に正しく、継続。

### 2-2. オッズの扱い【決定=両方使う】
- **予想オッズ（前日〜枠順段階）**: netkeibaの想定/AI予想オッズを o とし、自前の p と比較して**全レースの一次EV採点**を出す。アプリの主戦場。
- **確定オッズ（発走直前）**: 参戦判定が出たレースについて、**確定オッズで最終EVを再計算**し「割安のまま残っているか」を確定。ここが購入トリガー。
- 二段構え: 「予想オッズEVでスクリーニング → 確定オッズEVで最終ゲート」。予想オッズは変動する前提で、あくまで一次選別と割り切る。

### 2-3. 勝率 p の見立て — 段階的スコアリング【決定=数値EVまで出す】
全レース対象・数値EV精度重視のため、**言語分析に留めず段階的に確率モデル化する**。詳細は §3。
- フェーズ0（土台）: ルール取捨（新馬/未勝利・少頭数=減点）。既存 refresh.mjs のロジックを一次フィルタとして残す。
- フェーズ1（MVP）: claude-in-chrome が全レースを読み、馬ごとに**妙味スコア(0〜100)を数値化**（枠/脚質適性/斤量/騎手/前走位置取り/馬場適性/市場人気との乖離）。スコア→暫定確率 p に写像し、**数値EVを算出・表示**。
- フェーズ2（較正）: 検証ログ（実結果・実払戻）で「スコア帯 → 実勝率」の対応表を作り、p をキャリブレーション。EVの精度を継続的に引き上げる。

## 3. 数値EV算出の設計方針【新設・決定】

### 3-1. パイプライン（全レース共通）
```
出馬表/枠順/オッズ（取得）
  → 特徴量抽出（枠・脚質・斤量・騎手・前走・馬場・距離適性・人気）
  → 妙味スコア 0〜100（claude-in-chrome の構造分析＋ルール加点）
  → スコア→勝率 p へ写像（フェーズ1は暫定関数、フェーズ2は較正済み表）
  → EV = p × o − 1（券種別に単勝/複勝/ワイド/3連複を計算）
  → +EV/様子見/-EV の3段＋数値EVを併記して提示
```

### 3-2. p 写像の段階アプローチ（精度を上げる順）
1. **暫定写像（フェーズ1）**: スコアを単調変換した仮の p。「数値は暫定」と明示。まず動かすことを優先。
2. **人気補正**: 市場オッズが示す implied prob（控除込み）を基準線に、スコアの上振れ/下振れだけを p の補正に使う（市場を無視せず"乖離"だけ拾う）。
3. **較正（フェーズ2）**: 検証ログが N レース貯まったら、スコア帯ごとの実勝率でロジスティック較正。ここで数値EVが実用精度に。

### 3-3. 検証ログで較正するループ【決定=実損益まで追う】
- 毎開催、選出レース/馬について **賭け金・券種・想定/確定オッズ・結果・実払戻・損益** を記録（results.js 等）。
- 集計: 回収率（払戻/投資）、的中率、EV帯別の実回収（+EVと判定した群は本当にプラスか）。
- フィードバック: 「+EV判定なのに実回収が低いスコア帯」を検証部が検出 → EVロジック部が写像/加点を修正 → 次開催へ。**この閉ループがアプリの精度の心臓部**。
- 注意: p が未較正のフェーズ1では数値EVに「暫定」ラベルを必ず付け、過信を防ぐ（司令官の購入判断は最終ゲート＝確定オッズEV＋人の目）。

## 4. データ取得方式 — claude-in-chrome × GitHub Actions【決定=秘書代行】

**思想: 機械的に取れるものは Actions、判断が要るものは claude-in-chrome（秘書が代行）。**

| 工程 | 担当 | 備考 |
|---|---|---|
| 開催・出走一覧、クラス/距離/頭数/枠順の抽出、ルール一次フィルタ | **Actions（refresh.mjs 拡張）** | 既存が稼働中。全場・全レースへ拡張 |
| 予想オッズ/想定オッズの取得 | **Actions（API/スクレイピング。可否は§8）** | 定型取得 |
| 全レースの妙味スコア算出・特徴量読み取り | **claude-in-chrome（秘書代行＝頭脳）** | 全レース対象。§5で工数を現実化 |
| 予想オッズ×スコアで数値EV算出→参戦/見送り→買い目 | **claude-in-chrome（秘書代行）＋司令官レビュー** | EVの意思決定 |
| 確定オッズでの直前EV再計算・最終ゲート | **秘書代行＋司令官の購入判断** | 責任判断は自動化しない |
| PWA反映（data.js/auto.js 生成・push） | **Actions＋手動push併用** | 定型 |

- **秘書運用の前提（決定反映）**: claude-in-chrome は秘書(Claude)が起動・操作する。実行環境（秘書が使う端末/セッション）とnetkeibaログインアカウントは**秘書運用を前提に用意**。ログイン情報の管理・netkeiba会員種別（プレミアムだと直前オッズ取得の制約が緩む）は §8 で要決定。
- 二層構造は既存 index.html が対応済（KEIBA_AUTO優先表示）。**Actions=全レースの一次採点(auto.js)、claude-in-chrome=全レースのEV精査(data.jsへ昇格)** に役割拡張。

## 5. 運用フロー（全レースEVスコアリング前提）【具体化】

全場・全レースを対象にしつつ工数を現実化するため、**「Actionsで機械採点 → AIは差分と上位を精査」の二段構え**にする。

| タイミング | 司令官(PT) | 秘書(Claude/claude-in-chrome) | Actions | 部署 |
|---|---|---|---|---|
| 金曜夜(枠順前) | 対象開催・予算感を一言 | ― | **全場・全レースの出馬表取得＋ルール一次採点(auto.js)** | データ取得部 |
| 土 前日(枠順確定) | 上位候補をレビュー・GO指示 | **全レースを妙味スコア化→予想オッズで数値EV算出→+EV抽出→買い目下書き(data.js)** | 予想オッズ取得 | EVロジック部 |
| 日 前日 | 同上 | 同上（翌日分） | 同上 | EVロジック部 |
| 当日 朝 | 最終確認(15分) | 差分・+EVレースだけ提示、PWA更新→push | Pages再ビルド | フロント部 |
| 発走 直前 | **確定オッズEVで"割安のまま残るか"照合→購入** | 参戦レースのみ確定オッズ取得→EV再計算 | ― | ― |
| 開催後 | 結果・実払戻を一言 | **賭け金/払戻/損益をログ化→回収率更新** | 結果取得(将来) | 検証部 |

### 全レースを現実的な工数で回す工夫
- **一次フィルタで母数を削る**: Actionsのルール採点で「明確な-EV/見送り（新馬・未勝利・少頭数）」を先に落とし、AIが精査する母数を圧縮。全レースに"採点"は付くが、AIの深掘りは中位以上に集中。
- **バッチ処理**: claude-in-chrome の操作は場・レース単位でテンプレ化し、秘書が連続実行。人手の逐次判断を挟まない。
- **段階表示**: 前日は「全レースの暫定EV（予想オッズ）」、直前は「参戦レースの確定EV」。全レースに直前確定オッズは張らない（netkeiba取得回数制約・工数のため）。
- **司令官の関与は3点に圧縮**: 「金曜の一言」「前日レビュー」「直前照合＋購入」。全レース対象でも張り付き運用にはしない。

## 6. 画面/機能の方向性（残す/足す・MVP範囲）

### 残す（優秀な既存資産）
- PWA構成・競馬場テーマUI・カード開閉・検索フィルタ（index.html は流用）。
- 「今週の方針／買い方の方針／全レース判定(scan)／複勝転がし」の情報設計。
- data.js 差し替え運用、KEIBA_AUTO優先の二層構造。

### 足す（EV支援への進化）
1. **数値EVバッジ**: 各レース/馬に「p × o − 1 の数値EV ＋ +EV/様子見/-EV の3色」。フェーズ1は数値に「暫定」印。
2. **全レースEV一覧**: 現行 scan を拡張し、全レースにEVスコア列を追加。ソート/フィルタ（+EVだけ表示）。
3. **予想/確定オッズ欄**: 前日は予想オッズ、直前は確定オッズを差し込める（現行 odds.rows 流用）。
4. **検証ログ画面(results)**: 開催ごとの選出→賭け金→払戻→損益→通算回収率。**最重要**。

### MVPの範囲
- **In**: 全レースのEVスコア＋数値EV(暫定)、+EV抽出、買い目(本線＋5頭BOX)、予想オッズ表示、実損益を記録する検証ログ。
- **Out(後回し)**: 較正済みの高精度p（フェーズ2）、確定オッズ全自動取得、自動購入。
- 基準: **「全レースにEVが付き、+EVだけ抽出でき、実損益が記録できて回収率が見える」までがMVP**。

## 7. 部署編成【4部・着手順・依存関係】

| 部署 | ミッション（1行） |
|---|---|
| **データ取得部** | netkeibaから全場・全レースの出馬表/枠順/予想・確定オッズを安定取得し機械可読化する（Actions/refresh.mjs＋claude-in-chrome取得の下支え）。 |
| **EVロジック部** | 妙味スコア定義・p写像・EV算出を設計し、全レース採点と+EVの買い目を選出する（アプリの頭脳）。 |
| **フロント部** | PWAにEVバッジ・全レースEV一覧・オッズ欄・検証ログを実装しiPhoneで完結させる。 |
| **検証部** | 実損益・回収率を記録分析し、pをキャリブレーションしてEVロジック部に還元する。 |

### 着手順と依存関係
```
[1] データ取得部  ── 全レース＋オッズが取れないと何も始まらない（最上流）
        │  出力: 機械可読の出馬表＋オッズ
        ▼
[2] EVロジック部  ── 取得データを前提にスコア→p→EVを設計（データ取得部に依存）
        │  出力: EV採点済みデータ（data.js の中身）
        ▼
[3] フロント部    ── EV採点データを画面化（EVロジック部の出力仕様に依存）
        │  出力: PWA表示＋検証ログ入力口
        ▼
[4] 検証部        ── 運用開始後、実損益ログを分析しEVロジック部へ還元（[2]と閉ループ）
```
- **並行可能**: フロント部の器（EVバッジ枠・ログ画面UI）は、EVロジック部のデータ仕様が固まれば先行着手可。
- **クリティカルパス**: データ取得部 → EVロジック部。ここが遅れると全体が止まる。最優先。
- 当面の主役: **データ取得部＋EVロジック部**。フロント部はMVP実装フェーズ、検証部は運用開始後に本稼働。

## 8. 残論点（要検討として明記）と director への引き継ぎ

### 8-0. 司令官による最終決定（2026-07-05・残論点2件を決着）
- **アカウント方針**: netkeiba は**無料会員の範囲で運用**する（プレミアムは使わない）。無料で取れない情報（直前オッズ等）は**別サイトから補完取得**してよい。→ データ取得部は「無料netkeiba＋補完ソース」を前提に取得設計する。プレミアム前提の設計は不採用。
- **検証ログ入力**: **秘書（Claude）が代行入力**する。司令官の手入力・半自動は不要。→ 検証部/フロント部は秘書が入力しやすい導線（券種・金額・払戻の記録UI/バッチ）を前提に設計。

### 8-1. 決着済みだが実装前に潰す技術/規約リスク
- **オッズ取得の可否・規約**: netkeiba（無料範囲）/JRA/補完サイトのオッズ取得（スクレイピング）の利用規約適合とレート制限を、データ取得部が実装前に検証・報告。直前オッズは無料netkeibaで不足する場合の補完ソースを特定する。
- **秘書運用のログイン管理**: claude-in-chrome が使う端末/セッションとnetkeiba無料アカウント。資格情報の保管方法。
- **p未較正期の数値EVの見せ方**: フェーズ1は「暫定」ラベル必須。過信防止のUI文言。

### 8-2. director への引き継ぎ — 次に決めるべき要件定義項目
1. **妙味スコアの具体式**: 加点項目とウェイト（枠/脚質/斤量/騎手/前走/馬場/人気乖離）の初期値。
2. **スコア→p 写像関数**: フェーズ1の暫定関数の形（線形/ロジスティック）と人気補正の掛け方。
3. **対象券種と算出範囲**: 全レースでどの券種のEVまで出すか（単複ワイドは全馬、3連複は上位のみ 等）。
4. **オッズ取得の実装方式**: Actions取得 vs claude-in-chrome取得の切り分け、取得頻度、直前タイミング定義。
5. **データスキーマ**: data.js/auto.js/results.js の項目定義（EV・p・o・賭け金・払戻・損益）。
6. **検証ログの入力導線**: 秘書代行入力を前提に、記録UI/バッチの形（券種・金額・払戻・損益の入力導線）。
7. **MVPの完了定義(DoD)**: 「全レースEV表示＋+EV抽出＋実損益記録＋回収率表示」で1開催を通し運用できること。
8. **運用SLA**: 前日までにEV採点完了、直前◯分前までに確定EV提示、の時刻目標。
