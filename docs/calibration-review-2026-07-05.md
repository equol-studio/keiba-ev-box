## 1. H（人気乖離）因子 — 最優先で見直す

### 1-1. なぜ「5戦で人気馬が勝った」だけでは較正の根拠にならないか
- 人気馬が勝つのは競馬として普通（1番人気の複勝的中率は経験的に約6割前後）。5戦中数件「人気馬が好走してH減点と逆行した」こと自体は、この母集団なら起こりうる範囲であり、それだけでは「Hの重みが間違っている」と断定できない。
- **しかし今回はサンプル数の話ではなく、H算出手順（requirements.md §1-3）そのものの設計に欠陥がある**。これはロジックを読むだけで指摘できる、母数に依らない問題。

### 1-2. 設計欠陥の具体的箇所
requirements.md §1-3の手順:
```
乖離 = 予想人気順位 − 構造順位
乖離 ≤ -4 → H = -10
```
- これは「**構造スコア（A〜G）が低いのに市場人気が高い馬」を機械的に最大減点する**式。しかし構造スコアが低い理由は多様で、`A〜Gの採点漏れ・過小評価`によるものと`本当に不安要素がある`ものを区別できない。
- 特に今回のリハーサルはA(枠適性)/C(斤量)/D(騎手)が**3項目とも0点固定（データ未取得）**（rehearsal §スコープ）。この状態でS0（構造素点）を出すと、素点のレンジが本来より狭く縮んでおり、実力上位の人気馬でもS0が伸びにくい＝「構造順位が実力よりも下振れしやすい」状況になっていた。**その下振れしたS0を根拠にH=-10という最大級のペナルティを科す**ため、二重に不利が乗る設計になっている。
  - 函館12R④（市場1番人気、H=-10で大幅減点）→実際は優勝。函館9R①⑥⑬（人気馬、H=-10）→3着・5着と普通に好走。函館10R①（構造1位でもあり別ケース）は見送り自体は正しかった。
  - 共通パターン: **A/C/D未実装によりS0が痩せている中で、Hだけが±10のフルレンジで動く**ため、Hの相対的な影響力が設計時の想定より肥大化している。

### 1-3. 改善案（数値レベル）

**案A（推奨）: A/C/D未実装の間はHのレンジを暫定的に圧縮する。**
```
現行: 乖離 ≥ +4 → +10 / +2〜+3 → +5 / -1〜+1 → 0 / -2〜-3 → -5 / ≤-4 → -10
暫定: 乖離 ≥ +4 → +6  / +2〜+3 → +3 / -1〜+1 → 0 / -2〜-3 → -3 / ≤-4 → -6
```
理由: A/C/Dが0点固定でS0のレンジが本来(±31点相当)より狭い（A+8+C+5+D+6=+19点、マイナス側同様に-19点の可能性が失われている）。この分だけHの絶対値も比例して圧縮するのが整合的。**A/C/Dが実装されデータが入り次第、Hは requirements.md 原案の±10に復帰させる。** つまりこれは恒久的な弱化ではなく「データ欠損期間の暫定措置」と位置付ける。

**案B（保留・却下理由あり）: マイナス方向だけ非対称に弱める。**
- 「人気馬を過剰減点している」という観測だけを見ると、マイナス側だけ弱めたくなるが、**それは今回のサンプルの結果に直接フィットさせる過剰適合**であり、避けるべき。プラス側（市場が過小評価＝妙味）を temperature 高いまま残すと、逆に「妙味あり判定」側の精度も未検証のまま高い信頼度で使うことになり一貫性を欠く。**対称に縮める案Aを採用**。

### 1-4. 優先度
**最優先（K1のうちH該当箇所を今週中に修正）。** 理由: 5戦連続で+EV検出がほぼ0という状況で、Hの過剰減点は「本来+EVになり得た人気馬を機械的に潰している」可能性があり、サンプル不足を待たずに検出精度そのものを歪めている構造バグ。次回較正観測の母数を汚す前に直す価値が高い。

## 2. E（前走内容）— 重みではなく運用ルールを追加

- rehearsal発見: 函館10R⑦が前走16着大敗の一発評価のみでE=-10（＋人気乖離も連動して-10）となったが、実際は3着好走。
- **これはウェイト（+10〜-10のレンジ）の問題ではない。** 「前走の不利（出遅れ・詰まり・エンジン不利等）の有無」という入力情報を採点時に見ていないことが原因。requirements.md §1-2のE表には既に「不利（出遅れ/詰まり明記）ありの負け → +5」という行が存在するが、**運用（scoring-template.mdでの実際の確認手順）でこの行が使われた形跡がない**（今回のE=-10判定は「1.0秒差以上の大敗」の行だけを機械的に適用したと推測される）。
- **改善案**: K1の数値は変えない。代わりに docs/scoring-template.md に以下を追記する運用ルールを追加:
  1. 前走がE=-5または-10に該当する場合、**必ず該当レースのレース脚質コメント欄・上がり3F順位を確認し、不利の有無を判定してから加点行を選ぶ**（機械的な着順・着差だけで即決しない）。
  2. 前走が「大敗だが不利あり」の場合はE=+5の行を優先適用し、着差による-5/-10は不利がないことを確認できた場合のみ適用する。
- 優先度: **中**（次回較正サイクルまでに運用ルール文書を直せば十分。スコア式自体の変更ではないため実装コスト小）。

## 4. 触ってはいけない項目（サンプル不足で判断できない）

過剰反応（オーバーフィッティング）を避けるため、以下は5サンプルでは**動かさない**:

| 項目 | 理由 |
|---|---|
| K2 (p補正係数 α=0.30) | +EV検出が0〜1件では「α=0.30が強すぎる/弱すぎる」の判断材料がない。むしろ+EV検出が少ないのは市場効率が高いレースを選んだ結果である可能性が高く、αのせいと断定できない |
| K3 (EV閾値 +0.10/-0.05) | 同上。閾値を緩めれば+EV検出は増えるが、それは「基準を甘くしただけ」で精度向上ではない。閾値は据え置き、まず検出母数（N=100ベット）を貯めるのが先 |
| K4 (ワイド相関係数1.20) | 今回のリハーサルはワイドのEV算出（確定オッズ段のみ）を実施していない。触る根拠データが無い |
| K5 (Harville無補正) | 3連複は的中0件だが、そもそもBOXは「参考掲載」に格下げ済みで実購入対象でもない。補正の要否を判断する材料が現時点でない |
| K7 (EV帯の区切り) | 集計自体がまだ実施されていない（results.jsへの記帳はゼロ件）。帯の偏りを議論する母数がない |
| K8 (複勝p上限0.90) | 今回のサンプルでクリップに掛かった馬がいたか不明。該当なしなら判断材料なし |

**このリストの存在自体が今回のレビューの価値**: 「直すべき」と「まだ直せない」を混同すると、5戦の結果に無数のパラメータを同時にフィットさせてしまい、次回以降の較正シグナルをかえって読みにくくする。

## 6. 来週以降の運用（頻度・スコープ）具体案

司令官方針「全レース深掘りは重すぎると判明済み。少しずつ検証」を踏まえ、以下を提案。

### 6-1. スコープ：参戦候補レースに絞る（全レース深掘りはしない）
- **一次判定（ルールベース、Actions）は従来どおり全場・全レースに適用**（コスト実質ゼロ、継続）。
- **フルスコアリング（claude-in-chrome・秘書代行）は「参戦」判定が出たレースのみ**に絞る。「様子見」は当面対象外でよい（函館9R・12Rの様子見レースは今回、的中含め参考情報止まりで実購入価値が低かった）。
- 1開催あたりの参戦レース数は今回の実績で3場36レース中14レース（rehearsal §スコープ「参戦14」）。全レース深掘り（36レース）よりは軽いが、まだ多い場合は**開催のうち1〜2場に絞る**（例: 重賞・メインがある場を優先）運用も許容する。

### 6-2. 頻度：週1開催・段階的に対象を広げる
- **週1回（土日の開催）を基本単位**とし、いきなり「毎週全参戦レース」を目指さず、**まず週2〜4レース程度のフルスコアリングを継続**し、N=100ベット（フェーズ2較正のトリガー、requirements.md K1）に着実に近づける。
- ペース目安: 週3レース×継続 なら約8ヶ月でN=100に到達。もし司令官の時間が許せば週5〜6レースまで引き上げれば約4ヶ月に短縮。**この「週あたり何レース深掘りするか」自体を司令官と要相談**（後述の残論点）。

### 6-3. 較正の粒度：修正は月次でまとめる
- 毎週の个別レース結果で都度パラメータを弄らない（オーバーフィット防止）。**月1回、蓄積したレースをまとめてレビューし、H・E等の運用ルールを見直す**サイクルにする。今回のようなCEOレビューをその月次較正の場として使う。
- ただし今回のH因子の設計欠陥（§1）のように**「サンプル数によらず言える構造バグ」は即座に直す**。ロジックバグの修正と、データドリブンな数値較正（K2/K3等）は別トラックとして扱う。

### 6-4. 実行フロー（改訂）
```
土日開催 → Actions一次判定(全場) → 参戦レースのみ秘書がフルスコアリング
  → 前日EV記録 → (可能なら)直前確定オッズで再計算1〜2件試行
  → 開催後、実結果と照合してrehearsal形式でdocs/に記録
  → 月1回、蓄積分をCEOがレビューし較正要否を判断
```
