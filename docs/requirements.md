## 1. 妙味スコアの具体式【確定】

### 1-1. 式

```
妙味スコア S = 50（基礎点） + A + B + C + D + E + F + G + H
最後に 0〜100 にクリップ（S < 0 → 0, S > 100 → 100）
```

- 対象: 全レース・全出走馬（1頭につき1つのS）。
- 採点者: 秘書（claude-in-chrome で netkeiba 出馬表・馬柱を読み、下表の判定基準どおりに離散値を付ける）。判定に必要な情報が読めない項目は 0 点（中立）とし、独自解釈で埋めない。

### 1-2. 加点項目とウェイト初期値（全項目、離散値のみ。中間値は使わない）

| 項目 | 評価内容 | 判定基準（上から順に最初に該当した行を採用） | 加点 |
|---|---|---|---|
| A. 枠適性 | 当該コースの枠バイアス×馬番 | 有利枠（例: 福島芝の1〜3枠、小倉ダ1700の内中枠） | +8 |
| | | やや有利 | +4 |
| | | 中立 | 0 |
| | | やや不利 | -4 |
| | | 不利枠（例: 小回り多頭数の8枠、1角まで短い外枠） | -8 |
| B. 脚質適性 | コース・想定ペースと脚質の合致 | 合致（例: 前残りコースの逃げ先行） | +10 |
| | | やや合致 | +5 |
| | | 中立/不明 | 0 |
| | | やや不利 | -5 |
| | | 不利（例: 前残りコースの追込一辺倒） | -10 |
| C. 斤量 | 前走比の増減 | -2.0kg 以上減 | +5 |
| | | -1.0〜-1.5kg | +3 |
| | | ±0.5kg 以内 | 0 |
| | | +1.0〜+1.5kg | -3 |
| | | +2.0kg 以上増 | -5 |
| D. 騎手 | 当年リーディング順位＋乗り替わり | リーディング1〜10位 | +6 |
| | | 11〜30位 | +3 |
| | | 31位以下 | 0 |
| | | 上位騎手→31位以下への乗り替わり | -3 |
| E. 前走内容 | 着差・位置取り・不利の有無 | 勝ち or 0.2秒差以内 | +10 |
| | | 0.3〜0.5秒差 | +6 |
| | | 不利（出遅れ/詰まり明記）ありの負け | +5 |
| | | 掲示板（0.6秒差以上） | +3 |

| | | 実績なし/不明 | 0 |
| | | 想定馬場で6着以下のみ（2走以上） | -6 |

| | | 同コース3着内あり | +3 |
| | | 初コース/実績なし | 0 |
| | | 同コースで6着以下のみ（2走以上） | -5 |
| H. 人気乖離 | 構造評価順位 vs 市場人気順位 | （後述 1-3 の手順） | -10〜+10 |

- 理論レンジ: A〜G合計後の素点 50-42=8 〜 50+40=90、H を加えて -2〜100 → クリップで 0〜100。
- ウェイトはすべて**仮決め**。フェーズ2でスコア帯別実回収により調整する。

### 1-3. H. 人気乖離の算出手順（2パス方式）

1. まず H を除いた素点 S0 = 50+A+…+G をレース内全馬で計算し、S0 降順で「構造順位」を付ける。
2. netkeiba 予想オッズの単勝人気順位（予想人気）と比較し、`乖離 = 予想人気順位 − 構造順位`。
3. 乖離 ≥ +4 → H=+10 ／ +2〜+3 → +5 ／ -1〜+1 → 0 ／ -2〜-3 → -5 ／ ≤ -4 → H=-10。
   （プラス乖離＝市場が過小評価＝妙味。マイナス乖離＝過剰人気＝割高）
4. 予想オッズ未取得のレースは H=0 とし、data.js の当該レースに `"hPending": true` を付ける。

**【2026-07-05・CEO較正レビュー→解消】** A(枠適性)/C(斤量)/D(騎手)が未実装(常に0点)の間、
S0のレンジが狭く縮んでいるのにHだけがフルレンジ±10で効き、実力上位の人気馬を過剰減点する
構造欠陥があったため(docs/calibration-review-2026-07-05.md §1)、一時Hを±6に圧縮していた。
その後 A/C/D のデータ取得手段を整備した(下記)ため、上記3の原案(±10)に復帰。

- A(枠適性)/B(脚質適性): netkeiba `race/data_top.html?race_id=` の「コース分析」欄（無料）を使用。
  函館の主要コースは docs/hakodate-course-bias.md に取得済み。他場・他コースは都度確認。
- C(斤量前走比): scoring-template.md §1の手順どおり shutuba_past.html から現在斤量と前走斤量を比較。
- D(騎手): docs/jockey-leading-2026.md（JRA騎手リーディング、週次更新目安）を参照。

**次回スコアリングから必ずA/C/Dも実データで採点すること。** 0点固定のままHだけ±10に戻すと
過剰減点問題が再発するため、A/C/D未実装のまま運用しないこと。

## 2. スコア → p 写像関数【確定・フェーズ1暫定】

市場を基準線に「乖離だけ拾う」方式（ceo-direction §3-2 準拠）。

### 2-1. 単勝 p_win

```
step1  市場含意確率:  p_imp = 0.80 / o_win        （0.80 = 単勝払戻率）
step2  スコア補正:    m(S) = 1 + 0.30 × (S − 50) / 50   → [0.70, 1.30] にクリップ
step3  補正後:        p' = p_imp × m(S)
step4  レース内正規化: p_win_i = p'_i / Σ_j p'_j    （全馬合計 = 1）
```

- 補正係数 α = 0.30 は**仮決め**（S=100 で市場比 1.3倍、S=0 で 0.7倍）。
- o_win は一次採点時=予想オッズ、直前=確定オッズ。同じ式を両段で使う。

### 2-2. 複勝 p_place

```
step1  o_mid = (複勝オッズ下限 + 上限) / 2
step2  p_imp3 = 0.80 / o_mid
step3  p'' = p_imp3 × m(S)     （m は 2-1 と同一）
step4  正規化: p_place_i = 3 × p''_i / Σ_j p''_j   （全馬合計 = 3。8頭未満のレースは 2 ×）
step5  個別クリップ: p_place_i > 0.90 → 0.90
```

### 2-3. 3連複（5頭BOX）p_combo

- Harville 近似を使う。正規化済み p_win から、組 {a,b,c} が3着内を独占する確率:
```
P(a,b,c) = Σ over 6順列(i,j,k) of  p_i × p_j/(1−p_i) × p_k/(1−p_i−p_j)
p_combo(BOX) = 選定5頭から成る10組それぞれの P(a,b,c) の合計
```
- Harville は先行有利バイアス等を持つが、フェーズ1では補正しない（**仮決め**。フェーズ2で検証）。

### 2-4. EV と3段判定

```
EV_win   = p_win   × o_win  − 1
EV_place = p_place × o_mid  − 1
EV_wide  = p_place_A × p_place_B × 1.20 × o_wide − 1     （相関係数1.20は仮決め。確定オッズ段のみ算出）
EV_combo = p_combo × o_combo合成 − 1                      （確定オッズ段のみ算出）
3連複の前日提示 = 損益分岐合成オッズ BE = 1 / p_combo     （「合成オッズが BE を超えていれば +EV」）
```

3段判定（馬単位。EV_win と EV_place の高い方で判定。**閾値は仮決め**）:

| 判定 | 条件 |
|---|---|
| +EV | EV ≥ +0.10 |
| 様子見 | -0.05 ≤ EV < +0.10 |
| -EV | EV < -0.05 |

レース単位の call: 既存ルールフィルタ（新馬/未勝利/7頭以下=見送り）を先に適用し、通過レースのうち「+EV 馬が1頭以上」→ 参戦、「様子見馬のみ」→ 様子見、「全馬 -EV〜様子見未満」→ 見送り。

- フェーズ1の全数値EVに `provisional: true` を付け、UI では「暫定」印を必ず表示する（ceo-direction §8-1）。

## 3. 対象券種と算出範囲【確定】

| 券種 | 前日（予想オッズ段） | 直前（確定オッズ段） | 対象 |
|---|---|---|---|
| 単勝 | 数値EV算出 | 再計算 | 全レース・全馬 |
| 複勝 | 数値EV算出 | 再計算 | 全レース・全馬 |
| ワイド | 算出しない（予想段でワイドオッズ非提供のため） | 数値EV算出 | 参戦レースのスコア上位5頭の全組合せ10点 |
| 3連複 | p_combo と損益分岐合成オッズ BE のみ提示 | 実合成オッズで数値EV確定 | 参戦レースの上位5頭BOX（10点）のみ |
| 馬連・馬単・3連単・枠連 | 対象外（MVPのOut） | 対象外 | ― |

- 買い目提示形式: 本線（複勝 or ワイド）を主役とし、控除率の軽い複勝・ワイドを優先（CEO方針）。
- **3連複5頭BOXは「毎回必須」から変更【2026-07-05・司令官決定】**: T12リハーサル5レースで的中0/5・BOX選定(スコア単純上位5頭)も較正未了と判明したため、BOXは常時掲載する本線ではなく、**損益分岐合成オッズBEを実合成オッズが上回る（=EV_combo>0）場合のみ「参考掲載」**する任意項目に格下げする。全レースでBOXの p_combo/BE 計算自体は続ける（较正データ収集のため）が、司令官への提示は+EVの時のみに絞る。
- 直前の確定オッズ取得は**参戦判定レースのみ**（全レースには張らない。ceo-direction §5）。

## 4. オッズ取得の実装方式【確定】

### 4-1. 切り分け

| 対象 | 方式 | 頻度・タイミング |
|---|---|---|
| 出馬表・枠順・クラス・頭数（全場・全レース） | Actions（refresh.mjs 拡張） | 金 21:00 JST（既存cron）＋手動dispatch |
| 予想オッズ（単勝・複勝、全場・全レース） | Actions（refresh.mjs 拡張。取得不能時は秘書が claude-in-chrome で代替） | 金 21:00 JST ＋ 土日 7:00 JST（cron追加） |
| 確定オッズ（単勝・複勝・ワイド・3連複、参戦レースのみ） | claude-in-chrome（秘書代行） | 発走15分前に取得開始 |
| レース結果・払戻 | claude-in-chrome（秘書代行） | 開催当日夜〜翌日 |

### 4-2. 「直前」の定義

**発走時刻の15分前に確定オッズ取得を開始し、10分前までに最終EVを司令官へ提示する。** これより後のオッズ変動は追わない（購入判断は司令官）。

### 4-3. 取得元URLと補完サイト（無料範囲）

| 用途 | 第1候補 | 補完（第1が不可の場合） |
|---|---|---|
| 出馬表 | `race.netkeiba.com/top/race_list_sub.html?kaisai_date=` ＋ `race/shutuba.html?race_id=`（現行） | JRA公式 出馬表 |
| 予想オッズ | `race.netkeiba.com/odds/index.html?race_id={id}&type=b1`（非ログイン） | 秘書が claude-in-chrome で同ページを閲覧・転記 |
| 直前確定オッズ | **JRA公式オッズ**（jra.go.jp → オッズ。単複・ワイド・3連複を無料で提供） | Yahoo!スポーツナビ競馬 オッズ |
| 結果・払戻 | JRA公式 レース結果 | netkeiba 結果ページ |

- 直前オッズを netkeiba に頼らない理由: 無料会員ではオッズ更新間隔に制約があるため（ceo-direction §8-0 の「別サイト補完」= **JRA公式を第1補完に確定**。公式データかつ閲覧は規約上問題なし）。
- レート制限: Actions のリクエスト間隔 350ms（現行踏襲）。1レースあたりの取得回数上限 = 出馬表1回＋予想オッズ2回（金・当日朝）＋確定オッズ1回。
- 実装前検証タスク: 予想オッズページが fetch（非JS）で取得可能かを T2 の最初に確認。動的レンダリングで不可なら `race.netkeiba.com/api/api_get_jra_odds.html?race_id=` を検証し、それも不可なら「予想オッズは秘書の claude-in-chrome 転記」に切り替える（**仮決め: fetch可能と想定**）。
- fail-safe: 現行踏襲。パース成功レースが3未満なら auto.js を上書きしない。オッズのみ失敗時は出馬表部分だけ更新し `odds: null` とする。

## 6. 検証ログの入力導線【確定・秘書代行】

1. **入力者は秘書のみ**。司令官は開催後にチャットで一言報告するだけ（例: 「函館11R 複勝⑫500円→払戻1400円。3連複BOXは外れ、1000円」）。
2. 秘書は報告を §5-3 のエントリJSONに変換し、results.js の entries **先頭**に追記（新しい順）→ `git push origin main`。
3. 変換規約（docs/results-entry-template.md に手順書化。タスクT9）:
   - oddsYoso / evYoso / score は当該開催の data.js から転記（手入力しない）。
   - oddsFinal / evFinal は直前最終ゲート時の finalGate 記録から転記。記録がなければ `null`。
   - 払戻不明のエントリは作らない（司令官に払戻額を確認してから記帳）。
   - 同着・返還は result: "refund"、payout = 返還額。
4. 未購入（見送り）レースはエントリを作らない。「+EV判定したが見送った」レースの追跡はフェーズ2（MVPのOut）。

## 7. MVP完了定義（DoD）【確定】

以下7条件を**すべて**満たしたらMVP完了。1条件でも欠けたら未完了。

1. 1開催週末（土日2日・全場）の全レースに、妙味スコアと単勝・複勝の数値EV（「暫定」印付き）が PWA 上で表示される。
2. 全レースEV一覧（scan拡張）で「+EVのみ表示」トグルと EV降順ソートが動作する。
3. 参戦レースに、本線買い目＋損益分岐合成オッズが表示される。3連複5頭BOXはEV_combo>0の場合のみ参考掲載（§3参照、必須ではない）。
4. 参戦レースの直前確定オッズによる最終EV再計算が、発走10分前までに司令官へ提示された実績が2レース以上ある。
5. その週末の全購入が results.js に記録され、PWA 上で通算・券種別・スコア帯別・EV帯別の回収率と的中率が表示される。
6. §8 の SLA を、当該週末の全項目で達成している。
7. Actions が取得失敗した場合に auto.js が壊れない（fail-safe 動作をログで確認済み）。
